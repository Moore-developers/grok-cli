# Grok SKILL 能力路由方案

## 目标

设计一个以 OAuth 为主的 Grok SKILL，使用户能够通过自己的 OAuth 来使用
Grok 的完整能力。

这个 SKILL 的本质不是“复刻 Hermes 本身”，而是设计一套 Grok 能力路由层：

1. 用户提出“用 Grok 做某件事”。
2. SKILL 判断这件事属于哪一种 Grok 能力面。
3. 如果本地还没有可复用的 Grok OAuth 凭据，就先执行 Grok OAuth。
4. 认证成功后保存凭据，供后续复用。
5. 自动恢复并继续完成用户原始任务。

同时，这套设计要尽量参考 Hermes 当前在使用 Grok OAuth 时所依赖的参数、
值和行为特征，以提高兼容性和一致性。

## 定位

这不是一个单纯的“登录帮助器”，也不是一个只做聊天切换的 provider 配置。

它应该被设计成一个统一的 Grok 能力入口，使得用户只要表达“想用 Grok 做
什么”，SKILL 就能负责：

- 判断能力类型
- 判断是否需要先认证
- 发起认证
- 保存认证状态
- 恢复原始任务
- 把任务路由到正确的 Grok 能力面

## 范围

这份设计要覆盖的是 Hermes 当前已经接入的 Grok 能力面：

- 聊天 / 推理 / 工具调用
- X 搜索
- 图片生成
- 视频生成
- 文本转语音
- 语音转文字 / transcription
- OpenAI 兼容代理接入

这套设计明确采用 OAuth-first 策略。

默认情况下不应静默退回 API key 模式。只有在用户明确允许的前提下，
API key 才应作为异常分支存在。

## 依据来源

本方案基于 Hermes 仓库中的现有行为整理，重点参考：

- xAI Grok OAuth 指南
- 共享 xAI 凭据解析器
- xAI 的图片 / 视频 / TTS / STT 集成
- xAI 代理适配器
- x_search 工具说明
- Hermes 在 Grok OAuth 过程中使用的参数和行为特征

## 高层架构

未来的 Grok SKILL 建议围绕五层结构来设计：

1. 意图路由层
   负责把用户请求映射到某一种 Grok 能力面。

2. 认证闸门层
   负责判断本地是否已有可复用、且仍有效的 xAI OAuth 凭据。

3. OAuth 执行层
   当凭据缺失或失效时，负责执行浏览器 PKCE 认证流程。

4. 凭据持久化层
   负责保存、读取和复用凭据，避免下一次任务再次登录。

5. 能力分发层
   在认证成功后，恢复原始任务，并分发到正确的 Grok 能力面。

## 需要纳入路由的能力面

### 1. 聊天与推理

这是最核心的 Grok 运行路径。Hermes 把 Grok OAuth 当成一个 provider，
用于驱动标准聊天、推理、流式输出和工具调用。

### 2. X 搜索

Hermes 通过 Responses API 暴露 xAI 内建的 `x_search` 工具。这是一条独立
于普通网页搜索的能力面。

### 3. 图片生成

Hermes 通过 xAI 图像接口完成 Grok 图片生成，并复用同一套共享凭据解析器。

### 4. 视频生成

Hermes 通过 xAI 视频接口完成文生视频和图生视频，同样依赖共享凭据模型。

### 5. 文本转语音

Hermes 通过 `/v1/tts` 调用 xAI TTS，底层同样走共享 xAI 凭据解析。

### 6. 语音转文字

Hermes 通过 `/v1/stt` 调用 xAI transcription，同样复用共享 xAI 凭据。

### 7. 代理

Hermes 可以把 OAuth 登录后的 xAI 能力暴露成一个本地 OpenAI-compatible
proxy。这应该被看作一种单独的下游消费方式。

## SKILL 行为设计

当用户说“用 Grok 做 X”时，SKILL 应该遵循下面的行为：

1. 识别该请求明确指向 Grok。
2. 将任务归类到某一个 Grok 能力面。
3. 检查是否已有可复用的 Grok OAuth 凭据。
4. 如果凭据有效，则直接继续执行。
5. 如果凭据缺失或失效，则先执行 Grok OAuth。
6. 保存新的凭据。
7. 自动恢复并继续执行用户原始任务。

SKILL 不应该停在“登录成功”这一步。登录只是中间步骤，不是用户目标。

## 建议的意图分类

建议把用户请求内部映射成以下模式：

- `chat`
- `x_search`
- `image_gen`
- `video_gen`
- `tts`
- `stt`
- `proxy`

示例触发语句：

- 用 Grok 总结这个仓库
- 让 Grok 去 X 上搜一下大家的反应
- 用 Grok 生成一张图片
- 用 Grok 做一个短视频
- 用 Grok 把这段话说出来
- 用 Grok 把这段音频转成文字

## OAuth-first 原则

这套设计默认采用以下原则：

- 通过这个 SKILL 使用 Grok，意味着 Grok OAuth 是首选认证路径
- 已保存的 OAuth 状态应尽可能优先复用
- API key 不是默认路径
- 错误处理必须区分“需要重新登录”和“账号权限 / entitlement 被拒绝”

## 关于 Hermes 参数与值的参考方式

这套 SKILL 设计需要研究 Hermes 在 Grok OAuth 中用到的参数和值，但这些内容
在方案中应该被视为“兼容性输入”和“行为参考”，而不是 SKILL 的表面目标。

也就是说，这套设计关注的是：

- 哪些参数会影响 Grok OAuth 是否可用
- 哪些固定值可能是 Hermes 当前依赖的行为特征
- 哪些请求形状会影响 token 交换或 entitlement 判定
- 如何让用户自己的 OAuth 会话尽量保持与 Hermes 当前行为一致

## 建议的运行时状态机

```text
用户提出 Grok 任务
        |
        v
识别对应能力面
        |
        v
检查已保存的 OAuth 状态
        |
   +----+----+
   |         |
 有效      缺失/失效
   |         |
   v         v
执行任务   执行 OAuth 流程
   |         |
   |      保存状态
   |         |
   +----<----+
        |
        v
恢复原始任务
```

## 为什么应该做成 SKILL

这套行为非常适合做成 SKILL，因为它具备这些特点：

- 可重复使用
- 天然是多步骤流程
- 需要跨会话保持状态
- 它是能力驱动，而不是命令驱动
- 一致的恢复行为会显著提升体验
