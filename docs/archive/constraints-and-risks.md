# 约束与风险

## 目标

记录设计 Grok SKILL 能力路由、并接入用户自己的 OAuth 以使用 Grok 能力时，
必须面对的约束、假设和风险。

## 关键约束

### 1. OAuth 成功不等于能力一定可用

账号可能已经完成 OAuth，但在实际推理或 refresh 阶段仍被 xAI 的 entitlement
或 tier 规则拒绝。

### 2. OAuth 行为依赖上游假设

Hermes 看起来依赖了一组特定的 xAI OAuth 行为、请求形状、参数和值，而这些
假设未来可能被上游修改。

### 3. X 搜索对模型有依赖

即使 xAI 凭据有效，`x_search` 也仍然依赖当前模型是否支持该服务端工具。

### 4. 远程登录在工程上比较脆弱

最难排查的问题往往不是凭据无效，而是 callback 无法正确回到远程监听器。

### 5. 共享认证并不消除工具级配置

图片、视频、TTS、STT 和搜索各自仍可能需要额外的工具启用或配置，不是只要
有 OAuth 状态就一定可用。

## 设计风险

### 风险：静默回退会掩盖真实认证状态

如果 SKILL 悄悄退回到 API key 模式，用户将失去“当前确实走的是 OAuth-first
Grok 路径”的确定性。

### 风险：登录流程中断任务而不是恢复任务

如果实现把认证当成最终目标，而不是中间前置步骤，最后体验就不会像 Hermes。

### 风险：能力路由不完整

如果 SKILL 只处理聊天，而忽略 media / search / TTS / STT / proxy 的统一
路由，那它实际上并没有完成一个完整的 Grok SKILL 能力入口设计。

## 由这些风险推导出的设计决策

1. 保持 OAuth-first。
2. 持久化认证状态，供后续复用。
3. 自动恢复原始任务。
4. 明确区分 entitlement 错误与重新登录错误。
5. 把能力路由当成一等设计问题来处理。
