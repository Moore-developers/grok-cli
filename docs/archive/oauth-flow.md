# Grok OAuth 流程

## 目的

说明未来 Grok SKILL 应该如何把浏览器 OAuth 流程视为所有 Grok 能力的前置
闸门。

## 核心流程

```text
+------------------+
| 用户说：         |
| “用 Grok 去...”  |
+---------+--------+
          |
          v
+---------------------------+
| 是否触发 Grok 任务？      |
| chat / x_search / image   |
| video / tts / stt / proxy |
+-------------+-------------+
              |
              v
+----------------------------------+
| 检查已保存认证状态              |
| - 是否存在 xai-oauth            |
| - access token 是否存在         |
| - token 是否仍可用              |
+-------------+--------------------+
              |
      +-------+--------+
      |                |
      | 有效           | 缺失/失效
      v                v
+-------------+   +----------------------------+
| 复用 token  |   | 启动 xAI OAuth PKCE 流程   |
+------+------+   +-------------+--------------+
       |                        |
       |                        v
       |            +----------------------------+
       |            | 获取 discovery endpoints   |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 生成 PKCE verifier、       |
       |            | challenge、state、nonce   |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 选择回调模式               |
       |            | local / ssh / manual-paste|
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 打开或打印 authorize URL  |
       |            | 指向 accounts.x.ai        |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 用户登录并授权             |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 接收 callback code，       |
       |            | 并校验返回 state          |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 用 code 换取 tokens        |
       |            +-------------+--------------+
       |                          |
       |                          v
       |            +----------------------------+
       |            | 保存 tokens 供后续复用     |
       |            +-------------+--------------+
       |                          |
       +------------->------------+
                                  |
                                  v
                    +-----------------------------+
                    | 继续执行原始 Grok 任务      |
                    +-----------------------------+
```

## 回调模式

### 本地桌面模式

这是最理想的情况。浏览器在本机打开，loopback callback 也回到同一台机器。

### SSH 远端模式

授权页可能在本地浏览器中打开，但 callback listener 实际运行在远程主机上，
因此必须配合端口转发。

### 手动粘贴模式

当无法使用常规 SSH 隧道时，必须通过手动粘贴 callback URL 的方式继续流程。

## 对 SKILL 的设计要求

未来 SKILL 必须保留这条原则：

OAuth 是内部前置步骤，用户原始任务才是主任务。认证完成后，SKILL 必须自动
恢复并继续执行原任务。
